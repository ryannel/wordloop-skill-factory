## Primary Key Strategy

Choose the primary key type based on the distribution and exposure requirements of the entity.

### Decision Matrix

| Requirement | Recommended PK | Rationale |
|---|---|---|
| Internal-only, single-node | `BIGINT GENERATED ALWAYS AS IDENTITY` | Compact (8 bytes), monotonic append, excellent cache locality |
| Distributed / sharded / external API | `UUID` via `uuidv7()` (PG18+) or `gen_random_uuid()` | Globally unique, time-ordered (v7), no coordination required |
| Offline-first clients | `UUID` via `uuidv7()` | Clients can generate IDs without server round-trip |
| Time-series / event / log data | Natural key (`timestamp`, `device_id`) or no PK | Surrogate keys add overhead with no uniqueness benefit |

### BIGINT Identity (Default for Single-Node)

```sql
CREATE TABLE users (
    id    BIGINT GENERATED ALWAYS AS IDENTITY,
    email TEXT   NOT NULL,

    CONSTRAINT pk_users PRIMARY KEY (id)
);
```

- Prefer `GENERATED ALWAYS` over `GENERATED BY DEFAULT` — prevents accidental manual ID injection.
- 8 bytes, monotonic, maximizes B-tree page fill factor and cache locality.
- Sequence gaps are normal (rollbacks, crashes, concurrent transactions). Never try to make IDs consecutive.

### UUID v7 (Default for Distributed Systems)

```sql
CREATE TABLE workspaces (
    id         UUID        NOT NULL DEFAULT uuidv7(),  -- PG18+ native
    name       TEXT        NOT NULL,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now(),

    CONSTRAINT pk_workspaces PRIMARY KEY (id)
);
```

- UUIDv7 embeds a Unix timestamp in the most significant 48 bits → lexicographically sortable, near-sequential insert behavior.
- Reduces B-tree page splits from ~5,000 per 1M rows (UUIDv4) to ~10-20 per 1M rows.
- On PG17 and earlier, use `gen_random_uuid()` (v4) — random scatter is acceptable for low-to-moderate insert rates.
- UUIDv7 leaks creation time — if timestamp privacy is required, use a separate opaque external ID.

### What NOT to Use

- **`SERIAL` / `BIGSERIAL`:** Legacy syntax with ownership and permission quirks. Use `GENERATED ALWAYS AS IDENTITY` instead.
- **UUIDv4 for high-velocity inserts:** Random scatter causes index fragmentation. Use UUIDv7 or BIGINT.
- **Composite natural keys as PK on reference tables:** Use a surrogate PK and add a `UNIQUE` constraint on the natural key instead.

## Normalization Decisions

**Normalize first (to 3NF)** to eliminate data redundancy and update anomalies. Denormalize **only** for measured, high-ROI reads where join performance is proven problematic.

### When to Stay in 3NF

- The data is mutable (updated frequently). Denormalized copies become stale.
- Multiple access patterns exist (no single "hot query" dominates).
- Data integrity matters more than read latency.

### When to Denormalize

- A specific read path is proven slow via `EXPLAIN ANALYZE` and cannot be improved with indexes.
- The denormalized data is immutable or rarely updated (e.g., order snapshot at purchase time).
- Materialized views or generated columns can serve the use case without duplicating storage manually.

### Denormalization Techniques

| Technique | Use Case | Trade-off |
|---|---|---|
| Generated column (`STORED`) | Computed value from same row | Storage cost, auto-maintained |
| Generated column (`VIRTUAL`, PG18+) | On-the-fly computation, no storage | CPU cost per read, not indexable until stored |
| Materialized view | Pre-computed aggregate/join | Staleness until `REFRESH`, storage cost |
| JSONB sidebar column | Optional/sparse attributes | No FK integrity, GIN overhead |
| Redundant column + trigger | Cross-table denormalization | Trigger maintenance complexity |

## Naming Conventions

| Element | Convention | Example |
|---|---|---|
| Tables | `snake_case`, singular | `user`, `order_item` |
| Columns | `snake_case` | `created_at`, `workspace_id` |
| Primary key | `pk_{table}` | `pk_users` |
| Foreign key | `fk_{child}_{parent}` | `fk_orders_users` |
| Unique constraint | `uq_{table}_{columns}` | `uq_users_email` |
| Check constraint | `chk_{table}_{description}` | `chk_orders_positive_total` |
| Exclusion constraint | `excl_{table}_{description}` | `excl_bookings_no_overlap` |
| Index | `idx_{table}_{columns}` | `idx_orders_user_id` |

- **Never use quoted/mixed-case identifiers.** PostgreSQL lowercases unquoted identifiers. Quoted identifiers break tooling and create confusion.
- **Be explicit.** Anonymous constraints break migration diffing and `ALTER TABLE` operations.

## Relationship Modeling

### One-to-Many (1:N)

```sql
CREATE TABLE orders (
    id      BIGINT GENERATED ALWAYS AS IDENTITY,
    user_id BIGINT NOT NULL,

    CONSTRAINT pk_orders      PRIMARY KEY (id),
    CONSTRAINT fk_orders_users FOREIGN KEY (user_id)
        REFERENCES users (id) ON DELETE CASCADE
);

-- FK index — PostgreSQL does NOT auto-create this
CREATE INDEX idx_orders_user_id ON orders (user_id);
```

### Many-to-Many (M:N) — Junction Table

```sql
CREATE TABLE user_roles (
    user_id BIGINT NOT NULL,
    role_id BIGINT NOT NULL,

    CONSTRAINT pk_user_roles      PRIMARY KEY (user_id, role_id),
    CONSTRAINT fk_user_roles_user FOREIGN KEY (user_id)
        REFERENCES users (id) ON DELETE CASCADE,
    CONSTRAINT fk_user_roles_role FOREIGN KEY (role_id)
        REFERENCES roles (id) ON DELETE CASCADE
);

-- Reverse lookup index (role_id is not the leading PK column)
CREATE INDEX idx_user_roles_role_id ON user_roles (role_id);
```

### One-to-One (1:1)

```sql
CREATE TABLE user_profiles (
    user_id BIGINT NOT NULL,
    bio     TEXT,

    CONSTRAINT pk_user_profiles      PRIMARY KEY (user_id),
    CONSTRAINT fk_user_profiles_user FOREIGN KEY (user_id)
        REFERENCES users (id) ON DELETE CASCADE
);
```

Use 1:1 tables to separate hot columns from cold columns in update-heavy workloads. This reduces bloat on the main table.

### ON DELETE / ON UPDATE Actions

| Action | Behavior | Use When |
|---|---|---|
| `CASCADE` | Delete/update child rows automatically | Child has no meaning without parent (order items → order) |
| `RESTRICT` | Block parent delete if children exist | Defensive default — prevents accidental data loss |
| `SET NULL` | Nullify FK column in child | Child is meaningful independently (e.g., assigned_to on a task) |
| `SET DEFAULT` | Reset FK to default value | Rarely used — requires a meaningful default |
| `NO ACTION` | Same as `RESTRICT` but checked at transaction end | Useful with deferrable constraints |

**Never leave `ON DELETE` / `ON UPDATE` implicit.** The default (`NO ACTION`) is often not what you want. Be explicit.

## Soft Delete vs Hard Delete

### Hard Delete (Preferred Default)

Use standard `DELETE` unless there is a concrete requirement for data recovery or audit.

- Keeps table size optimal, no index bloat.
- Simple queries — no `WHERE deleted_at IS NULL` everywhere.
- If history is needed, use an audit log table (see below).

### Soft Delete (When Required)

Use when the domain requires an "undo" or "recycle bin" feature visible to end users.

```sql
ALTER TABLE meetings ADD COLUMN deleted_at TIMESTAMPTZ;

-- Active-only queries use this partial index
CREATE INDEX idx_meetings_active
    ON meetings (workspace_id, created_at DESC)
    WHERE deleted_at IS NULL;

-- Unique constraint that only applies to non-deleted rows
CREATE UNIQUE INDEX uq_meetings_slug_active
    ON meetings (workspace_id, slug)
    WHERE deleted_at IS NULL;
```

**Rules for soft delete:**
- Always add partial indexes with `WHERE deleted_at IS NULL` on every query path.
- Always add partial unique constraints that exclude deleted rows.
- Implement a periodic cleanup job to hard-delete rows past the retention window.
- Never rely on application-layer filtering alone — the database must enforce exclusion.

## Audit Log Patterns

### Trigger-Based Audit Log (Recommended)

```sql
CREATE SCHEMA IF NOT EXISTS audit;

CREATE TABLE audit.logged_actions (
    id              BIGINT GENERATED ALWAYS AS IDENTITY,
    schema_name     TEXT           NOT NULL,
    table_name      TEXT           NOT NULL,
    action          TEXT           NOT NULL,  -- INSERT, UPDATE, DELETE
    row_id          TEXT           NOT NULL,  -- PK value as text
    old_data        JSONB,
    new_data        JSONB,
    changed_fields  TEXT[],                    -- columns that changed (UPDATE only)
    actor_id        TEXT,                      -- from session variable
    action_at       TIMESTAMPTZ    NOT NULL DEFAULT clock_timestamp(),

    CONSTRAINT pk_audit_logged_actions PRIMARY KEY (id)
);

-- Partition by month for retention management
-- CREATE TABLE audit.logged_actions_2026_01 PARTITION OF audit.logged_actions
--     FOR VALUES FROM ('2026-01-01') TO ('2026-02-01');

CREATE INDEX idx_audit_table_action
    ON audit.logged_actions (table_name, action_at DESC);
```

```sql
-- Generic audit trigger function
CREATE OR REPLACE FUNCTION audit.log_changes() RETURNS TRIGGER AS $$
BEGIN
    INSERT INTO audit.logged_actions (schema_name, table_name, action, row_id, old_data, new_data, actor_id)
    VALUES (
        TG_TABLE_SCHEMA,
        TG_TABLE_NAME,
        TG_OP,
        CASE TG_OP
            WHEN 'DELETE' THEN (OLD).id::text
            ELSE (NEW).id::text
        END,
        CASE WHEN TG_OP IN ('UPDATE', 'DELETE') THEN to_jsonb(OLD) END,
        CASE WHEN TG_OP IN ('INSERT', 'UPDATE') THEN to_jsonb(NEW) END,
        current_setting('app.actor_id', true)
    );
    RETURN NULL;  -- AFTER trigger, return value ignored
END;
$$ LANGUAGE plpgsql;
```

```sql
-- Attach to a table
CREATE TRIGGER trg_users_audit
    AFTER INSERT OR UPDATE OR DELETE ON users
    FOR EACH ROW EXECUTE FUNCTION audit.log_changes();
```

**Best practices:**
- Use `AFTER` triggers, not `BEFORE` — audit the committed state.
- Use `clock_timestamp()` not `now()` — `now()` returns transaction start time.
- Set `app.actor_id` via `SET LOCAL app.actor_id = '...'` at the start of each request.
- Partition the audit table by time for efficient retention (`DROP TABLE` of old partitions is instant).
- Store old/new data as `JSONB` so a single audit table works for all business tables.

## Temporal Tables

For "time-travel" queries where you need to reconstruct entity state at any point in time.

```sql
-- Main table with system period
CREATE TABLE products (
    id         BIGINT GENERATED ALWAYS AS IDENTITY,
    name       TEXT        NOT NULL,
    price      NUMERIC     NOT NULL,
    sys_period TSTZRANGE   NOT NULL DEFAULT tstzrange(now(), NULL),

    CONSTRAINT pk_products PRIMARY KEY (id)
);

-- History table (same structure minus defaults)
CREATE TABLE products_history (
    id         BIGINT      NOT NULL,
    name       TEXT        NOT NULL,
    price      NUMERIC     NOT NULL,
    sys_period TSTZRANGE   NOT NULL,

    CONSTRAINT excl_products_history_no_overlap
        EXCLUDE USING GIST (id WITH =, sys_period WITH &&)
);

CREATE INDEX idx_products_history_lookup
    ON products_history (id, sys_period);
```

Use a trigger to move the old version to `_history` on every `UPDATE` or `DELETE`, closing the `sys_period` range. Query history with range containment: `WHERE sys_period @> '2026-01-15T12:00:00Z'::timestamptz`.

## Table Types

| Type | Durability | Use Case |
|---|---|---|
| Regular (`CREATE TABLE`) | Fully durable, WAL-logged | All production data |
| `UNLOGGED` | Persistent but not crash-safe | Caches, staging, rebuildable data |
| `TEMPORARY` | Session-scoped, auto-dropped | Scratch work, intermediate results |

- **UNLOGGED tables** skip WAL logging → 2-5x faster writes. Data is lost on crash. Use for caches or materialized staging that can be rebuilt.
- **TEMPORARY tables** are invisible to other sessions. Ideal for complex multi-step queries within a single request.

## Generated Columns

### Stored (PG12+)

Computed at write time, persisted on disk, indexable.

```sql
CREATE TABLE profiles (
    id         BIGINT GENERATED ALWAYS AS IDENTITY,
    first_name TEXT NOT NULL,
    last_name  TEXT NOT NULL,
    full_name  TEXT GENERATED ALWAYS AS (first_name || ' ' || last_name) STORED,

    CONSTRAINT pk_profiles PRIMARY KEY (id)
);

CREATE INDEX idx_profiles_full_name ON profiles (full_name);
```

### Virtual (PG18+)

Computed at read time, no storage cost, not directly indexable.

```sql
ALTER TABLE products
    ADD COLUMN price_with_tax NUMERIC
    GENERATED ALWAYS AS (price * 1.19) VIRTUAL;
```

Use virtual columns for display-only computations. Use stored columns when you need to index the computed value.

## Anti-Patterns

| Anti-Pattern | Why It's Bad | Better Alternative |
|---|---|---|
| EAV (Entity-Attribute-Value) tables | Extremely complex joins, poor performance | JSONB for flexible attributes, typed columns for known fields |
| Random UUIDv4 for high-velocity PKs | B-tree fragmentation, 500x more page splits vs sequential | UUIDv7 or BIGINT |
| `SELECT *` in application queries | Fetches unnecessary columns, breaks when schema changes | Explicit column lists |
| OFFSET pagination on large tables | Performance degrades linearly with page depth | Keyset (cursor) pagination |
| Over-normalization for read-hot paths | Excessive joins on every request | Generated columns, materialized views |
| Implicit FK actions | Unclear cascade behavior, surprising data loss | Always specify `ON DELETE` / `ON UPDATE` |
